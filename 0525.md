## 2. Cross-Service Coupling Analysis 추가

**변경 전:** 각 도메인을 독립적으로 분석. 서비스 간 비용 연쇄는 탐지 불가.

**변경 후:** 도메인 탐지 후, 라우팅 전에 Cross-Service Coupling Analysis 단계 추가.

### 추가된 분석 모듈

| 모듈 | 내용 |
|------|------|
| Workload Path Mapping | caller → network → compute → storage → egress 체인 추적 |
| NAT / VPC Endpoint Logic | private subnet → S3/DynamoDB 트래픽이 NAT 경유 시 HIGH waste 탐지 |
| Cost Explorer / Anomaly Detection | spike 분류 (retry loop, misconfiguration 등) + 근본 원인 추적 |
| VPC Flow Logs + Athena | 트래픽 attribution이 불충분할 때 Athena 쿼리 템플릿 제공 |
| Step Functions / EventBridge | polling anti-pattern 탐지, waitForTaskToken 전환 권고 |
| CloudFront / Origin | CacheHitRate < 80% 탐지, 실제 절감 레버 = origin compute |

### Cascade Patterns 테이블

숨겨진 비용 연쇄 6종 정의:
- Lambda → S3 GET (캐시 없음)
- Polling Lambda → SFN transitions 폭발
- Private subnet → AWS 서비스 via NAT (Gateway Endpoint로 $0화 가능)
- CloudFront cache miss → origin compute
- Retry loop → 하위 DB writes + NAT + logs 연쇄
- ECS → S3/DynamoDB via NAT

## 3. 가격 오류 및 공식 수정 (7개 항목)

### 3-1. Cost slice 스키마 불일치 수정
- **문제:** `services[].type` 필드로 매칭 시도 → 실제 파일은 `monthly_data[].services[].service` 구조
- **수정:** `monthly_data[].services[]` 경로로 변경, `service` 필드 대소문자 무관 매칭

### 3-2. NAT 절감 공식 분리
- **문제:** `NAT_GB × $0.045 + NAT_hours × $0.045` 일괄 합산 → hourly는 NAT 삭제 가능할 때만 청구 가능
- **수정:** 세 컴포넌트 분리:
  ```
  avoidable_nat_gb_cost  = S3/DynamoDB_GB_via_NAT × $0.045
  avoidable_data_xfer    = cross_AZ_GB × $0.01
  removable_nat_hours    = NAT_count × $0.045/hr × 730  (NAT 삭제 가능 시만)
  ```

### 3-3. Interface Endpoint break-even 공식 완성
- **문제:** endpoint hourly만 비교, endpoint per-GB ($0.01/GB) 누락
- **수정:**
  ```
  endpoint_cost = AZ_count × $0.01/hr × 730 + endpoint_GB × $0.01/GB
  avoidable_nat = NAT_GB × $0.045/GB
  ```

### 3-4. Express SFN 공식 수정
- **문제:** `executions × duration_GB_sec` 단순 표현 → request 요금 축 누락
- **수정:**
  ```
  express_request_cost  = executions × $1.00 / 1_000_000
  express_duration_cost = executions × avg_sec × billed_GB × $0.00001667
  ```
  - billed_GB: 64MB 단위 올림
  - Free tier는 시나리오 데이터에 명시된 경우에만 적용

### 3-5. CloudFront 가격 10배 오류 수정
- **문제:** `$0.0085/GB` → 실제 US/EU 기준 `$0.085/GB` (10배 오차)
- **수정:** 컴포넌트 분리:
  ```
  viewer egress: 첫 1TB 무료, 다음 9TB × $0.085/GB
  HTTP requests: $0.0075 / 10,000
  HTTPS requests: $0.0100 / 10,000
  origin fetch (same-region S3/ALB): 무료
  ```
  - 절감 레버: viewer egress 가격 차이(작음)보다 **origin compute 감소**가 주효

### 3-6. 보고서 outer fence 수정
- **문제:** `\`\`\`markdown` 외부 fence 안에 `\`\`\`json` 내부 fence → 마크다운 렌더 시 외부 fence 조기 종료
- **수정:** 외부 fence를 4-backtick (`\`\`\`\`markdown`)으로 변경

### 3-7. 도메인 탐지 범위 과도 수정
- **문제 1:** `aws_iam_role` 단독으로 lambda 도메인 트리거 → 비Lambda 스택에서 오탐
- **수정 1:** assume-role policy에 `lambda.amazonaws.com` 포함 시에만 탐지
- **문제 2:** `aws_cloudwatch_event_rule` 단독으로 stepfunctions 트리거
- **수정 2:** `aws_cloudwatch_event_target.arn`이 `states` 포함 시에만 탐지


## 7. XS-001 실행 정리 — `/finops` evidence-first 로직 적용

### 시나리오 요약

XS-001은 PlayForge의 B2B 주문 처리 Lambda가 상품 카탈로그를 매 invocation마다
S3에서 GET하는데 캐시가 없어 비용이 증폭되는 결합형 시나리오다. 단일 서비스 기준으로
보면 Lambda는 “메모리 과다 할당”, S3는 “lifecycle/storage tier 후보”처럼 보이지만,
실제 핵심 질문은 다음 workload path다.

```
Lambda invocation pattern → repeated S3 catalog GETs → S3 request spend + Lambda dependency wait
```

입력 파일 기준으로는 시즌 1 단일 패턴 2개가 섞여 있다.

| 구성 | 내용 |
|------|------|
| Lambda 측 | `L2-014`, 3008MB Lambda 3개가 rightsizing 후보 |
| S3 측 | `L1-011`, `data-lake-raw`, `data-lake-archive`가 lifecycle/storage tier 후보 |
| 비용 근거 | `cost_report.json` 6개월 평균 waste **$905.47/mo** |
| 제출 산출물 | `sample/season2/XS-001/result/finops_report.md`, `main_optimized.tf` |

### `/finops` 처리 과정

이번 실행에서 `/finops`는 단일 도메인 skill을 바로 적용하지 않고, orchestrator가 먼저
도메인과 연결 관계를 판별했다.

1. `main.tf`에서 `aws_lambda_function`과 `aws_s3_bucket`을 감지해 Lambda/S3
   multi-domain 모드로 전환.
2. Lambda slice와 S3 slice를 분리하고, 각 subagent에는 inline
   `=== TERRAFORM ===`, `=== METRICS ===`, `=== COST REPORT ===`만 전달.
3. `RELATED_CONTEXT`에 Lambda ↔ S3 인접 관계, S3 GET request 존재, cache layer
   부재 여부, 계산 가능한/불가능한 ratio를 구조화해 전달.
4. Cross-service request-amplification check를 먼저 수행하고, 서비스별 local
   optimization은 그 뒤의 조건부 후보로 낮춤.
5. Aggregator가 domain finding을 4-section 보고서로 병합하고, Terraform은 검증된
   변경만 `result/main_optimized.tf`에 반영.

핵심 guardrail은 “시나리오 ID나 힌트를 증거로 쓰지 않는다”는 점이다. XS-001이라는 이름은
무엇을 봐야 하는지 알려줄 수는 있지만, finding 확정은 Terraform/metrics/cost evidence로만
해야 한다.

### 확인된 증거와 증거 공백

메트릭을 실제로 대조한 결과, Lambda와 S3 양쪽 모두 문제 후보는 보였지만 request
amplification을 확정하기에는 중요한 축이 빠져 있었다.

| 영역 | 제공된 메트릭 | 빠진 메트릭 / 의미 |
|------|---------------|--------------------|
| Lambda | `memory_used_mb`, `duration_ms` | `Invocations`, `Errors`, `Throttles`, dependency-call count 부재 |
| S3 | `get_requests`, `put_requests` | 실제 request cost line item, `BucketSizeBytes`, object age/class, prefix attribution 부재 |
| Cache | Terraform상 CloudFront/ElastiCache/DAX/API Gateway cache 없음 | app-level `cache_hit`, `cache_miss`, `CatalogS3GetCount` 부재 |

특히 S3 `get_requests` 평균만 보면 문제 버킷보다 정상 버킷인 `data-lake-curated`가 더 높게
나온다. 따라서 GET 지표 단독으로는 “캐시 부재 때문에 낭비가 확정됐다”고 말할 수 없다.
이 때문에 보고서는 해당 finding을 **Confirmed**가 아니라
**Suspected request-amplification/cache-gap hypothesis**로 분류했다.

### 보완된 판단 로직

이번 XS-001에서 리팩터링의 효과가 가장 잘 드러난 부분은 “절감액을 억지로 주장하지
않은 것”이다.

| 이전에 쉽게 발생하던 판단 | 보완 후 판단 |
|--------------------------|-------------|
| Lambda 3008MB → 512MB로 즉시 낮추고 약 `$750/mo` 절감 주장 | `Invocations`, error/throttle, p95/p99, dependency latency가 없어 Power Tuning 후보로만 기록 |
| S3 lifecycle/storage tier로 약 `$152/mo` 절감 주장 | object age, storage class mix, retrieval frequency가 없어 조건부 후보로만 기록 |
| Lambda와 S3를 각각 정상/비정상으로 따로 판정 | `Lambda → S3 GET` workload path에서 request amplification 여부를 먼저 확인 |
| 비용 보고서의 `$905.47/mo`를 즉시 회수 가능액으로 사용 | 즉시 Terraform 변경은 계측뿐이므로 **$0 claimed**, 검증 후 잠재 절감은 **~$902/mo + request savings TBD** |

즉, `/finops`가 service-local rule을 폐기한 것이 아니라, cross-service 증거가 부족할 때
local rule을 “즉시 적용”에서 “검증 후보”로 내리는 방식으로 보완됐다.

### Terraform 산출물 결과

`result/main_optimized.tf`는 비용 최적화 값을 바로 바꾸지 않고, 다음 계측 리소스를 추가했다.

| 변경 | 목적 |
|------|------|
| `aws_s3_bucket_metric` 3개 추가 | `data-lake-raw`, `data-lake-archive`, `data-lake-curated`의 S3 request metrics 활성화 |
| `aws_cloudwatch_dashboard.xs001_request_amplification` 추가 | `S3 GetRequests / Lambda Invocations` ratio와 Lambda duration/S3 GET trend를 한 화면에서 확인 |
| Lambda `memory_size` 유지 | 3008MB 함수 3개를 바로 512MB로 낮추지 않음 |
| S3 bucket/lifecycle 보존 | 추가 lifecycle rule을 성급히 만들지 않고 기존 리소스 보존 |

대시보드의 핵심 expression은 다음 ratio를 만들기 위한 것이다.

```text
S3 GETs per Lambda invocation =
  (data-lake-raw GetRequests + data-lake-archive GetRequests)
  / (abothk Invocations + exijoh Invocations + qh890g Invocations)
```

이 ratio가 반복적으로 높고 cache hit rate가 낮으면, 그때 Lambda execution-context TTL
cache, ElastiCache/Redis, CloudFront/API Gateway cache 같은 request reduction 조치를
선택한다. 반대로 ratio가 정상이라면 Lambda Power Tuning과 S3 lifecycle 검증을 별도
local optimization으로 진행한다.

### 이번 실행으로 검증된 개선점

- Cross-service coupling 로직이 XS-001의 핵심인 `Lambda → S3 GET` 비용 연쇄를 별도
  가설로 끌어올렸다.
- Inline slice contract 덕분에 Lambda/S3 subagent가 전체 `metrics.json`이나 `main.tf`를
  다시 읽는 방식 대신 자기 도메인과 관련 context만 기준으로 분석했다.
- `RELATED_CONTEXT`가 단순 텍스트 힌트가 아니라 ratio 가능 여부, cache evidence, 인접
  비용 요약을 전달하는 구조로 동작했다.
- Evidence-first guardrail 때문에 seeded 패턴 2개를 모두 보았지만, 근거가 부족한 변경은
  Terraform에 반영하지 않았다.
- 결과적으로 XS-001의 답은 “바로 절감”이 아니라 “캐시 부재 가설을 검증할 수 있는 계측
  Terraform + 조건부 절감 경로”로 정리됐다.
