# FinOps CloudWatch Metric Alarm Analysis Skill

## Scope

Reduce unnecessary custom-metric cost by downgrading alarms from high-resolution
(`period = 1`, $0.90/metric/month) to standard (`period = 60`, $0.30/metric/month)
when sub-minute granularity has no operational benefit.

## Input Mode

- **Orchestrator subagent mode** (`=== TERRAFORM ===` present): inline slices only.
- **Standalone mode**: scan `WORK_DIR` for full inputs.

Missing facts → write `Not available in the provided data; verify in the real environment.`

## Detection Rules (apply `rules/high_resolution_alarm.json`)

| Rule | Condition | Severity | Action |
|------|-----------|----------|--------|
| M1 | `resolution_seconds = 1` (or `period = 1`) AND `actual_required_resolution_seconds >= 60` | HIGH | DOWNGRADE_TO_STANDARD |
| M2 | `metric_type = "high_resolution"` with eval period >= 1 min and no sub-minute SLA | HIGH | DOWNGRADE_TO_STANDARD |
| M3 | Large high-resolution alarm count with no spikes needing sub-minute detection | MEDIUM | REVIEW_HIGH_RESOLUTION |
| M4 | Scenario-only attributes present (`resolution_seconds`, `actual_required_resolution_seconds`, `metric_type`, `evaluation_period_minutes`) | MEDIUM | NORMALIZE_TERRAFORM |

## Safety Guardrails

- Valid Terraform attribute is `period` (seconds); treat scenario-only attributes
  as evidence and normalize on output.
- Do not downgrade alarms with sub-minute SLA, anomaly detection on 1-s data,
  or real-time dashboards depending on 1-s granularity.
- AWS service namespace alarms are already standard resolution — not affected.

Full safety check list: `references/details.md` § Resolution Downgrade Safety Check.
