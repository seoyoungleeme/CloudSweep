# FinOps SQS Analysis Skill

## Scope

Reduce unnecessary API requests and retry waste while preserving latency,
consumer throughput, message visibility, FIFO ordering, DLQ behavior, and
application timeouts.

## Input Mode

- **Orchestrator subagent mode** (`=== TERRAFORM ===` present): inline slices only.
- **Standalone mode**: scan `WORK_DIR` for full inputs.

| File | Used For | If Missing |

## Detection Rules (apply `rules/short_polling_sqs.json`)

| Rule | Condition | Severity | Action |
|------|-----------|----------|--------|
| Q1 | `receive_wait_time_seconds = 0` and empty receives high | HIGH | ENABLE_LONG_POLLING |
| Q2 | Long polling enabled but empty receives remain high | MEDIUM | REVIEW_CONSUMER_POLLING |
| Q3 | Receive/delete/send volume suggests batching opportunity | MEDIUM | REVIEW_BATCHING |
| Q4 | High retries, DLQ movement, or message age drives repeated receives | MEDIUM | FIX_RETRY_VISIBILITY |
| Q5 | Client timeout or latency constraints unknown | INFO | VALIDATE_CLIENT_TIMEOUTS |

## Safety Guardrails

- Long polling is usually a strong cost optimization, but do not blindly set
  every queue to 20 s without checking consumer HTTP/read timeouts, latency
  requirements, Lambda ESM behavior, FIFO throughput, and backlog/age metrics.
- Do not assume long polling eliminates all empty receives; model partial reduction.

Full pre-change checklist: `references/details.md` § Required Safety Checks.
