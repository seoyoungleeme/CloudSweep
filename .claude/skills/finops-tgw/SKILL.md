# FinOps Transit Gateway Analysis Skill

## Scope

> **Review-only authority:** LangGraph owns detection and calculations. Review transitive routing, inspection, and multi-account requirements; do not rerun rule arithmetic.

Reduce avoidable TGW data processing and attachment charges without breaking
network connectivity, security segmentation, routing policy, or multi-account
architecture requirements.

## Detection Rules (apply `rules/tgw_rightsizing.json`)

| Rule | Condition | Severity | Action |
|------|-----------|----------|--------|
| T1 | TGW data processing on same-region VPC-to-VPC traffic that could use VPC Peering (esp. same-AZ free path) | HIGH | REPLACE_TGW_WITH_VPC_PEERING |
| T2 | TGW with <= 3 VPC attachments for simple point-to-point; no transitive routing, multi-account, inspection requirement | MEDIUM | EVALUATE_VPC_PEERING_REPLACEMENT |
| T3 | Attachment charge persists for attachments with no/minimal traffic | MEDIUM | REVIEW_IDLE_ATTACHMENT |
| T4 | Traffic routed cross-AZ through TGW when same-AZ TGW/peering path exists | MEDIUM | REVIEW_AZ_LOCAL_ROUTING |

## Safety Guardrails

- Do not recommend TGW → VPC Peering without confirming topology. TGW is
  required for hub-spoke multi-account, transitive routing, centralized
  inspection (Network Firewall, third-party appliances), SD-WAN/Direct Connect.
  VPC Peering does not support transitive routing — one peering per VPC pair.
- Treat existing `aws_vpc_peering_connection` resources as compliant/decoy
  unless inputs explicitly flag them as problematic.
- Do not delete TGW in one `terraform apply` — route changes must precede attachment deletion.

Full pre-change checklists: `references/details.md` § Required Safety Checks.
