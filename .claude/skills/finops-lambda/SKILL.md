# FinOps Lambda Analysis Skill

## Scope

> **Review-only authority:** LangGraph owns detection and calculations. Review duration, errors, retries, and dependency impacts; do not rerun rule arithmetic.

Reduce unnecessary compute, request, provisioned-concurrency, ephemeral-storage,
and retry/error cost without increasing latency, throttling, cold-start risk,
or failure rates.

## Detection Rules (apply `rules/overprovisioned_lambda.json`)

| Rule | Condition | Severity | Action |
|------|-----------|----------|--------|
| L1 | Memory over-allocated; modeled lower tier reduces total cost without latency/error risk | HIGH | RIGHTSIZE_MEMORY |
| L2 | Timeout greatly exceeds p99 duration; timeout/error path costly | MEDIUM | REDUCE_TIMEOUT |
| L3 | Provisioned concurrency enabled but utilization low / schedule too broad | HIGH | TUNE_PROVISIONED_CONCURRENCY |
| L4 | Ephemeral storage above free baseline without usage evidence | MEDIUM | REVIEW_EPHEMERAL_STORAGE |
| L5 | x86 function may be cheaper on arm64; deps support it | LOW | MODEL_ARM64 |
| L6 | Errors, retries, throttles, or event-source backlog drive cost | MEDIUM | FIX_ERROR_RETRY_COST |
| L7 | Downstream storage/API calls per invocation are high or rising; no cache evidence | HIGH | REDUCE_DEPENDENCY_CALLS_OR_CACHE |

## Safety Guardrails

- Do not downsize memory from average memory alone. Memory also controls
  CPU/network — lower memory can increase duration enough to erase savings.
  Use p95/p99 duration, max memory, errors, throttles, cold starts, and cost modeling.
- Memory reduction is not linear — recalculate with modeled duration at the new tier.
- When Lambda duration/cost co-moves with downstream request rate, investigate
  dependency-call amplification before recommending memory-only rightsizing.
  If invocation or downstream-call metrics are missing, report the gap and add
  instrumentation rather than treating the scenario as proven.
