## 1. MA-001 FinOps 설계 과정

### 1-1. 시나리오 설명

MA-001은 Season 2 Week 2 과제로 받은 **멀티 에이전트 도입 검증용 결합 시나리오**다. 회사는 PixelStorm이고, 상황은 B2C 주문 처리 인프라를 인수한 뒤 Lambda, S3, DynamoDB 비용 낭비가 한 Terraform 파일 안에 섞여 있는 상태다.

이 과제의 핵심은 단순히 비용 낭비를 찾는 것이 아니라, **단일 에이전트 구조가 결합 인프라에서 어떤 패턴을 놓치는지 측정하고, 멀티 에이전트 구조가 recall을 얼마나 회복하는지 증명하는 것**이다.

| 항목 | 내용 |
|------|------|
| 과제 | Season 2 Week 2 · MA-001 |
| 테마 | 멀티 에이전트 아키텍처 도입 |
| 회사/워크로드 | PixelStorm · B2C order processing |
| 제출물 | Analysis, Optimized Terraform, Estimated Monthly Savings, report.md |

시나리오에는 시즌 1에서 다뤘던 세 개의 독립 FinOps 패턴이 동시에 심어져 있다.

### 1-2. 신경 쓴 점

#### (a) 데이터 근거 엄수

분석 초기에 서브에이전트들이 원본 데이터에 없는 수치를 생성하는 현상이 있었다. 이를 방지하기 위해 각 수정 사항을 원본 파일로 직접 검증했다.

**Lambda 메트릭 검증**

최초 보고서는 "720 datapoints all flat 100, min = max = p99 = 100 MB, zero variance"로 기술했다. 그러나 실제 metrics.json을 확인하니 각 함수에 100 미만 datapoint가 1개씩 존재했다.

| 함수 | p99 | 실제 min |
|------|-----|---------|
| atrjum | 100 MB | **91.21 MB** |
| 37781v | 100 MB | **82.39 MB** |
| luzj7m | 100 MB | **86.75 MB** |

"flat"이라는 표현은 부정확하다. 올바른 기술은 "p99 100 MB, low variance."

**WCU 평균 재계산**

최초 보고서의 WCU avg = 84.9 (8.5% 활용률)는 근거가 불명확했다. 720개 datapoint를 Node.js로 직접 계산한 결과:

```
WCU avg = 87.94, p99 = 100, min = 0, utilization = 8.8%
RCU avg = 100.0, p99 = 100, min = 100, utilization = 2.0%
```

결론에는 영향이 없지만 보고서 수치는 실측값 기준으로 수정했다.

**S3 per-bucket GB 부재**

최초 보고서에서 "data-lake-raw 2,867 GB → $38/mo 절감, archive 5,325 GB → $101/mo 절감"이라고 기술했다. 그러나 metrics.json의 S3 리소스 섹션을 확인하면 `get_requests` 데이터만 존재하고 storage size(GB) 측정값은 없다. per-bucket 수치는 근거 없는 추정이었으므로 cost_report의 aggregate "$152/mo" 만 사용하도록 수정했다.

**DynamoDB pricing_note 모순 처리**

cost_report의 pricing_note는 "미사용 WCU 900 × $0.00065 × 730 ≈ $427, 미사용 RCU 4,500 × $0.00013 × 730 ≈ $427, 합계 약 $522"로 기술되어 있다. 그런데 $427 + $427 = $854이지, $522가 아니다. pricing_note 자체가 모순이다. 보고서에서 이 산술식을 그대로 반복하지 않고 "cost_report.json states ~$522/mo as the scenario-provided conservative estimate; raw unit-rate arithmetic is treated as an upper-bound cross-check only"로 처리했다.

#### (b) Lambda 과금 모델 오류 수정

"96.7% of allocated memory goes unbilled"라는 표현은 완전히 반대다. Lambda는 실제 사용량이 아니라 **할당 메모리(GB-sec) 기준으로 과금**된다. 미사용 용량도 과금된다. 올바른 표현: "96.7% of allocated memory is wasted billed capacity — billed but unused."

#### (c) decoy 보존

data-lake-curated(9m71do)는 `is_problem: false`이고 원본 main.tf에 `transition-rule`(90d → GLACIER)이 이미 존재한다. 최초 main_optimized.tf에서 이 버킷에 `noncurrent_version_expiration`과 MPU abort를 추가했는데, 이는 근거 없는 변경이었다. 원본 lifecycle 그대로 유지하도록 수정했다.

#### (d) MCP 가용성 처리

분석 시점에 aws-pricing MCP의 AWS credentials가 만료 상태(`InvalidClientTokenId`)였다. 이를 분석 블로킹 이유로 삼지 않고, pricing priority에 따라 cost_report.json pricing_note를 primary로 사용했다.

### 1-3. 멀티 에이전트 설계

#### 디스패치 흐름

```
finops orchestrator
│
├── main.tf 읽기 → 도메인 감지: lambda (comp1_), s3 (comp2_), dynamodb (comp3_)
│
├── [병렬] Agent tool 3개 동시 발사
│     ├── Lambda 전문가
│     │     TF chunk: comp1_ 5개 함수
│     │     Metrics slice: lambda 리소스 7,957 tokens
│     │     Cost slice: Lambda 항목 + pricing_note
│     │
│     ├── S3 전문가
│     │     TF chunk: comp2_ 3개 버킷 + 기존 lifecycle
│     │     Metrics slice: s3 리소스 5,744 tokens
│     │     Cost slice: S3 항목 + pricing_note
│     │
│     └── DynamoDB 전문가
│           TF chunk: comp3_ 2개 테이블
│           Metrics slice: dynamodb 리소스 3,341 tokens
│           Cost slice: DynamoDB 항목 + pricing_note
│
└── 결과 집계
      - finops_report.md (4개 통합 섹션: Problem Identification / Root Cause / Proposed Solution / Savings)
      - main_optimized.tf (3 컴포넌트 병합)
```

각 서브에이전트 프롬프트에는 검증된 수정 사항을 명시적으로 전달했다. 에이전트가 잘못된 수치를 재생성하지 않도록 "verified data corrections" 섹션을 프롬프트에 포함했다.

#### 산출 결과

| 도메인 | 패턴 | 발견 리소스 | 월 절감 |
|--------|------|-----------|---------|
| Lambda | L2-014 | 3 / 5 함수 | $750 |
| S3 | L1-011 | 2 / 3 버킷 | $152 |
| DynamoDB | L1-010 | 1 / 2 테이블 | $522 |
| **합계** | 3 패턴 | **6 리소스** | **$1,424/mo** |

Recall: **3 / 3 (100%)**. 단일 에이전트 베이스라인은 Lambda만 분석해 1/3 recall(33%)이었다.

#### MA-001 산출물 반영 확인

`sample/season2/MA-001/result/finops_report.md`와 `main_optimized.tf`를 기준으로 다시 대조했을 때, `0518.md`에 반영해야 할 핵심 결과는 다음과 같다.

| 확인 항목 | 반영 상태 |
|-----------|-----------|
| 분석 도메인 | Lambda, S3, DynamoDB 3개 모두 반영 |
| 발견 패턴 | L2-014, L1-011, L1-010 모두 발견 |

| 예상 절감액 | $750 + $152 + $522 = **$1,424/mo** |
| 원본 낭비 추정과의 차이 | cost_report 평균 낭비 $1,435.67/mo 대비 $11.67 차이. 보수적 산정과 반올림 차이로 허용 가능 |
| 측정값 한계 | 결과 파일의 `Total tokens`, `Wall-clock time`, `Analysis cost`는 `Not measured`로 남아 있음 |

Optimized Terraform에도 핵심 조치가 정상 반영됐다. Lambda는 문제 함수 3개의 `memory_size`가 3008MB에서 256MB로 줄었고, S3는 raw/archive 버킷에 lifecycle이 추가됐으며 curated 버킷의 기존 90일 Glacier 전환은 보존됐다. DynamoDB는 `dynamodb-table-wouiv4`의 기본 capacity가 5000/1000에서 120/120으로 조정됐고, Auto Scaling target은 현재 Terraform 산출물 기준 `min_capacity = 100`, `max_capacity = 500`이다. 따라서 문서에서는 "초기 capacity 120/120, Auto Scaling 범위 100-500"으로 구분해 적어야 한다.

대조 중 확인한 세부 불일치도 있다. `finops_report.md`의 Immediate Actions는 Auto Scaling min을 120으로 설명하지만, `main_optimized.tf`에는 min 100으로 작성되어 있다. 제출 기준 파일은 Terraform이므로 `0518.md`에는 실제 적용 파일의 값인 100-500을 기록하고, 필요하면 별도 후속 수정에서 report와 Terraform을 120-500 또는 100-500 중 하나로 맞춘다.

다만 제출 과제의 보너스 항목인 토큰 사용량과 wall-clock은 실제 LLM 실행 telemetry가 남아 있지 않다. 아래 토큰 섹션은 **API 실측값이 아니라 파일 크기와 도메인 슬라이스 크기 기반 추정치**로 구분해서 기록한다.

### 1-4. 보고서 포맷 재설계

#### 변경 이유

초기 finops_report.md는 에이전트(도메인)별로 섹션이 분리되어 있었다.

```
## finops-lambda Analysis   → Problem / Evidence / Root Cause / Solution / Savings
## finops-s3 Analysis       → ...
## finops-dynamodb Analysis → ...
## Cross-Domain Interactions
## Total Savings Summary
```

이 구조는 에이전트의 경계를 그대로 보고서에 노출한다. 독자 입장에서 "지금 당장 무엇을 해야 하는가?"를 파악하려면 세 섹션을 모두 읽고 스스로 통합해야 한다. Savings Summary도 끝에 있어서 전체를 읽기 전까지 규모 파악이 안 된다.

#### 새 구조

```
1. Problem Identification  — 모든 도메인 문제 리소스를 하나의 테이블로, 메트릭 증거 첨부
2. Root Cause             — 서비스 간 패턴과 cross-domain interaction을 단일 내러티브로
3. Proposed Solution      — 배포 순서(rollback risk 낮은 순) 기준 전체 액션 목록
4. Estimated Monthly Savings — 도메인별 breakdown + 총합
```

서비스 경계가 아니라 **분석가의 의사결정 흐름** 기준으로 정보를 재조직했다. 에이전트가 몇 개든 독자는 항상 같은 4개 섹션을 본다.

#### 구현 방법

SKILL.md의 Step 3 (Aggregate results)에 "Unified Report Structure" 섹션을 명시했다. 서브에이전트는 여전히 도메인별 findings 데이터를 반환하고, 오케스트레이터가 이를 4개 섹션으로 통합한다. 서브에이전트 프롬프트 구조는 변경하지 않았다 — 출력 형식만 aggregation 단계에서 바뀐다.

#### 설계상 유의점

**도메인 출처 추적**

Problem Identification 테이블에 Service 컬럼을 두어 어느 도메인에서 온 finding인지 식별 가능하게 했다. 섹션 제목에서는 "finops-lambda" 같은 표현을 제거하지만 독자는 Service 컬럼으로 출처를 알 수 있다.

**배포 순서 기준 정렬**

Proposed Solution의 Immediate Actions는 도메인별이 아니라 배포 순서로 정렬했다: DynamoDB → Lambda → S3.

| 순서 | 도메인 | 이유 |
|------|--------|------|
| 1 | DynamoDB | Auto Scaling 추가는 additive change — rollback이 가장 쉬움 |
| 2 | Lambda | memory 변경은 deploy + observe cycle 필요 |
| 3 | S3 | lifecycle 적용은 billing 반영에 24–48h 소요 |

Cross-Domain interaction(Lambda 버스트 → DynamoDB 스로틀링 위험)은 Root Cause 섹션 내에 인라인으로 포함했다. 별도 섹션이 아니라 인과관계가 설명되는 위치에 자연스럽게 배치하는 것이 독자 흐름에 맞다.

## 2. Orchestrator-Worker vs Routing — 본 도구에 어느 쪽이 맞나?

### 두 패턴의 차이

| | Routing | Orchestrator-Worker |
|-|---------|---------------------|
| 결정 시점 | 디스패치 전 단 한 번 | 실행 중 동적으로 |
| 작업 수 | 1 | N (병렬 또는 순차) |
| 결과 합산 | 없음 (단일 handler가 전부) | 있음 (orchestrator가 집계) |
| 상태 공유 | 없음 | orchestrator가 보유 |
| 예시 | if/switch | MapReduce, scatter-gather |

### 본 도구의 실제 패턴

CloudSweep는 엄밀히 말해 Orchestrator-Worker도 Routing도 아니다. **Static Fan-out + Aggregation**이다.

- Routing과 다른 점: 하나를 선택하는 게 아니라 모든 감지된 도메인에 디스패치한다.
- Orchestrator-Worker와 다른 점: 워커가 실행 중에 오케스트레이터와 통신하지 않는다. 오케스트레이터는 디스패치 전에 모든 결정을 완료한다.

패턴을 코드로 표현하면 이렇다:

```python
# CloudSweep의 실제 패턴
domains = detect_domains(main_tf)          # 디스패치 전 전체 결정
results = parallel_map(analyze, domains)   # 독립 실행 (통신 없음)
report = aggregate(results)                # 오케스트레이터가 집계
```

이는 MapReduce의 map 단계에 해당한다. 워커 간 메시지 패싱, 동적 라우팅, 중간 결과 기반 재디스패치가 없다.

### 재분류: Static Fan-out + Aggregation은 Orchestrator-Worker의 정적 변형이다

앞서 CloudSweep를 "Orchestrator-Worker도 Routing도 아닌" 패턴으로 기술했지만, 이는 구현 메커니즘(정적 디스패치)과 패턴 계층(Orchestrator-Worker)을 혼동한 것이다.

Orchestrator-Worker 패턴의 핵심 조건은 세 가지다:

| 조건 | CloudSweep 충족 여부 |
|------|---------------------|
| 오케스트레이터가 태스크를 분해하고 위임한다 | ✓ finops skill이 도메인 감지 후 Agent tool로 위임 |
| 워커는 특화된 실행 단위다 | ✓ lambda/s3/dynamodb 전문가 에이전트 |
| 오케스트레이터가 결과를 수집·통합한다 | ✓ 4-section unified report 생성 |

"실행 중 오케스트레이터와 통신하지 않는다"는 제약이 아니라 단순화다. MapReduce의 map 단계도 워커가 오케스트레이터와 중간 통신을 하지 않지만, 이를 두고 MapReduce가 Orchestrator-Worker가 아니라고 하지는 않는다. CloudSweep는 **정적 디스패치(Static dispatch)** 를 쓰는 Orchestrator-Worker다 — 디스패치 결정이 런타임이 아닌 사전에 이루어진다는 의미다.

```
Orchestrator-Worker 스펙트럼
│
├── Static dispatch    ← CloudSweep 현재 위치
│     도메인을 main.tf 파싱 시점에 결정
│     워커 수와 유형이 실행 전에 확정
│
└── Dynamic dispatch   ← 확장 시 이동할 방향
      중간 결과에 따라 워커를 추가로 생성
      오케스트레이터가 실행 중 판단
```

### 추후 확장에서 Orchestrator-Worker 구조가 유리한 이유

현재를 Orchestrator-Worker로 명확히 분류해두면, 다음 확장이 구조 변경 없이 이루어진다.

**확장 시나리오 1 — 동적 도메인 추가**

Lambda 에이전트가 DLQ로 SQS를 발견했을 때, 오케스트레이터가 SQS 도메인 분석을 추가로 디스패치하는 패턴이다.

```python
# 현재: 정적
domains = detect_domains(main_tf)          # [lambda, s3, dynamodb]
results = parallel_map(analyze, domains)

# 확장: 동적
domains = detect_domains(main_tf)
results = {}
while domains:
    batch_results = parallel_map(analyze, domains)
    results.update(batch_results)
    domains = orchestrator.expand_domains(batch_results)  # 추가 도메인 발견 시
```

워커 계약(입력 → findings)은 동일하다. 오케스트레이터 로직만 바뀐다.

**확장 시나리오 2 — 순차 의존성**

S3 분석 결과의 데이터 볼륨을 Lambda 에이전트의 비용 추정에 참조해야 하는 경우, 오케스트레이터가 S3 완료 후 Lambda를 디스패치하는 구조가 필요하다. 현재 구조에서는 오케스트레이터가 결과를 보유하므로 이 흐름 제어를 추가하는 것이 자연스럽다.

**확장 시나리오 3 — 에이전트 풀링**

도메인 수가 늘어날 때(10+ 도메인) 오케스트레이터가 동시 실행 수를 제한하고 큐잉하는 방식이 필요하다. Orchestrator가 중앙 조율자이므로 이 조율 로직을 오케스트레이터에만 추가하면 된다.

**결론**: "Static Fan-out + Aggregation"은 구현 명세고, **Orchestrator-Worker**가 패턴 명칭이다. 현재 CloudSweep는 Orchestrator-Worker의 정적 변형이며, 이 분류가 추후 확장의 자연스러운 진화 경로를 열어준다.

## 3. 컨텍스트 격리가 의미 있는 시점

### MA-001 파일 기반 토큰 추정값

`result/finops_report.md`에는 `Total tokens`, `Wall-clock time`, `Analysis cost`가 `Not measured`로 남아 있다. 따라서 아래 값은 실제 LLM API telemetry가 아니라, 파일 크기와 도메인별 슬라이스 크기를 기준으로 한 컨텍스트 규모 추정값이다.

```
파일                     크기        추정 토큰
───────────────────────────────────────────
main.tf (전체)           209 lines    ~315
cost_report.json         166 lines    ~2,500
metrics.json (전체)      291,890 B    ~76,813

metrics.json 도메인별 슬라이스:
  Lambda (5 resources)             ~7,957
  S3     (3 resources)             ~5,744
  DynamoDB (2 resources)           ~3,341
  합계                             ~17,042
```

### 격리의 의미가 있는 조건

**metrics.json은 이미 60K tokens를 초과한다 (76,813).** 따라서 전체 파일을 에이전트에 그대로 넘기면 context 상한을 압박한다. 반면 main.tf(315 tokens)와 cost_report.json(2,500 tokens)은 도메인 격리를 하지 않아도 된다.

컨텍스트 격리의 효과는 두 가지다:

**효과 1 — 토큰 절감**

| 시나리오 | 입력 토큰 합계 |
|---------|--------------|
| 단일 에이전트, 전체 metrics.json | ~76,813 + 2,815 = ~80K |
| 3 에이전트, 전체 metrics.json 각각 전달 | ~80K × 3 = ~240K |
| 3 에이전트, **도메인 슬라이스 전달** | ~17,042 + (2,815 × 3) = ~25,490 |

도메인 슬라이스 전달은 전체 파일을 각 에이전트에 전달하는 것보다 **10배 적은 토큰**을 사용한다. 심지어 단일 에이전트(80K)보다도 적다.

**효과 2 — 노이즈 제거**

S3 에이전트가 720개 Lambda memory datapoint를 처리할 필요가 없다. 도메인 슬라이스는 에이전트가 자기 리소스에만 집중하게 해 분석 정확도를 높인다.

### 격리가 의미 없는 조건

- main.tf가 작을 때 (315 tokens). 전체를 넘겨도 무방.
- cost_report.json이 작을 때 (2,500 tokens). 전체를 넘겨도 무방.
- metrics.json이 없거나 아주 작을 때 (< 10K tokens). 격리 불필요.

### 격리의 세 차원

컨텍스트 격리는 "어떤 데이터를 넘기느냐" 하나가 아니다. 세 차원이 있다.

| 차원 | 정의 | 현재 CloudSweep 상태 |
|------|------|---------------------|
| **데이터 격리** | 각 에이전트가 자기 도메인에 필요한 데이터만 수신 | 부분 구현 |
| **상태 격리** | 에이전트 간 findings가 집계 전까지 독립 | 아키텍처상 자동 보장 |
| **도구 격리** | 각 에이전트가 자기 도메인에 필요한 도구만 접근 | 미구현 |

### 현재 모델이 수행하는 격리

**데이터 격리 (부분 구현)**

| 파일 | 격리 방식 | 상태 |
|------|----------|------|
| main.tf | comp1\_, comp2\_, comp3\_ 청크로 분리 전달 | ✓ 구현됨 |
| metrics.json | 도메인별 리소스 슬라이스 추출 후 전달 | ✓ 구현됨 |
| cost_report.json | 전체 파일을 모든 에이전트에 동일하게 전달 | ✗ 미격리 |

cost_report.json은 현재 2,500 tokens로 작아서 문제가 없다. 하지만 도메인 수가 늘어나거나 cost_report가 커지면 격리 대상이 된다.

**상태 격리 (아키텍처상 자동 보장)**

Agent tool로 생성된 서브에이전트는 각자 독립적인 컨텍스트 윈도우를 가진다. Lambda 에이전트의 findings가 S3 에이전트의 컨텍스트에 유입되는 경로가 없다. 오케스트레이터가 결과를 수집하는 집계 단계에서만 findings가 합산된다. 이는 의도한 설계가 아니라 Agent tool 아키텍처의 기본 속성이다.

**도구 격리 (미구현)**

현재 모든 에이전트가 aws-pricing MCP, aws-docs MCP 등 전체 도구에 접근 가능하다. Lambda 에이전트가 S3 pricing을 조회하는 것을 막는 메커니즘이 없다. 이는 지금 규모에서는 문제가 되지 않지만, 에이전트 수가 늘어나면 불필요한 도구 호출이 발생할 수 있다.

### 확장 방법

**확장 1 — cost_report 도메인 슬라이싱**

cost_report.json이 커지거나 도메인별 pricing_note가 명확히 분리될 때 적용한다.

```python
def slice_cost_report(cost_report: dict, domain: str) -> dict:
    return {
        "monthly_cost": cost_report["monthly_cost"],
        "services": [s for s in cost_report["services"] if s["type"] == domain],
        "pricing_note": cost_report["pricing_notes"].get(domain, {}),
    }
```

**확장 2 — 도구 격리 (프롬프트 수준)**

Agent tool은 도구 목록을 동적으로 제한하는 인터페이스를 제공하지 않으므로, 프롬프트 수준에서 명시하는 방식을 쓴다.

```
# Lambda 에이전트 프롬프트에 추가
Available tools for this analysis:
- aws-pricing: AmazonEC2, AWSLambda service codes only
- aws-docs: Lambda, EC2 documentation only
Do NOT query S3 or DynamoDB pricing — those are handled by separate agents.
```

이는 강제(enforcement)가 아니라 지시(instruction)이지만, 불필요한 도구 호출을 줄이는 데 실질적으로 효과적이다.

**확장 3 — findings 스키마 정의**

에이전트가 텍스트 문자열 대신 구조화된 findings를 반환하면, 오케스트레이터의 집계 로직이 단순해지고 findings 간 교차 오염을 방지할 수 있다.

```python
# 현재: 에이전트가 markdown 텍스트 반환
findings: str = agent_result

# 확장: 에이전트가 구조화된 데이터 반환
findings: DomainFindings = {
    "domain": "lambda",
    "resources": [...],
    "monthly_savings": 750,
    "actions": [...],
    "cross_domain_signals": ["DLQ → SQS 도메인 가능성"]
}
```

cross_domain_signals 필드를 두면 오케스트레이터가 동적 도메인 확장 결정을 내릴 때 명시적 근거를 가질 수 있다 — Section 2에서 설명한 동적 Orchestrator-Worker 확장의 구현 전제 조건이다.

**결론**: CloudSweep에서 컨텍스트 격리는 metrics.json에서만 유의미하다. main.tf와 cost_report.json은 전체를 넘겨도 토큰 부담이 없다. 대용량 metrics 파일을 다루는 시나리오(L2 이상)에서 도메인 슬라이싱은 필수다. 확장 방향은 cost_report 슬라이싱 → findings 스키마 정의 순으로, 도구 격리는 에이전트 수가 충분히 늘어날 때 고려한다.

## 4. 15× 토큰 비용이 정당화되려면

### 추정 토큰 배수

위 추정값을 기준으로 하면 MA-001의 토큰 배수는 다음과 같다:

| 방식 | 총 입력 토큰 | 총 비용 (est.) |
|------|------------|--------------|
| 단일 에이전트 (full metrics) | ~80K | ~$0.24 |
| 3 에이전트 (도메인 슬라이스) | ~25K | ~$0.075 |
| 3 에이전트 (full metrics 각각) | ~240K | ~$0.72 |

도메인 슬라이싱을 제대로 하면 멀티 에이전트가 오히려 **단일 에이전트보다 저렴하다.** "15×"는 슬라이싱 없이 전체 metrics.json을 각 에이전트에 보낼 때 나오는 worst case다.

### 15× 비용이 정당화되는 가치 metric

만약 슬라이싱 없이 240K 토큰을 쓰는 경우라도, FinOps 도구에서 정당화 기준은 다음과 같다.

**핵심 metric: 증분 recall × 연간 절감액 / 분석 비용**

```
단일 에이전트 recall: 1/3 → $750/mo 발견
멀티 에이전트 recall: 3/3 → $1,424/mo 발견

증분 절감: ($1,424 - $750) × 12 = $8,088/yr
분석 비용 증가: $0.72 - $0.24 = $0.48 (15× worst case 기준)

ROI = $8,088 / $0.48 ≈ 16,850× 투자 대비 수익
```

FinOps 도구는 분석 비용이 발견된 절감액의 0.001%에 불과하므로 토큰 배수 자체는 의사결정 기준이 되기 어렵다. 오히려 정당화가 **실패하는** 조건을 정의하는 것이 더 유용하다.

**15× 비용이 정당화되지 않는 조건:**

| 조건 | 이유 |
|------|------|
| 모든 도메인이 동일 패턴을 가질 때 | 에이전트 수를 늘려도 recall이 증가하지 않음 |
| 비용 낭비가 한 도메인에 집중될 때 | 나머지 에이전트의 분석 가치가 낮음 |
| 메트릭 데이터가 없을 때 | 에이전트를 나눠도 분석 품질이 올라가지 않음 |
| 도메인 수가 1개일 때 | 라우팅으로 충분; 멀티 에이전트 오버헤드만 발생 |

**실용적 결론**: CloudSweep에서 토큰 비용 최소화보다 중요한 것은 슬라이싱이다. 슬라이싱을 하면 멀티 에이전트가 단일 에이전트보다 저렴하면서도 recall이 높다. 15× 배수 자체를 걱정할 필요가 없어진다.

## 5. Claude Subagent vs Strands vs LangGraph vs AgentCore

### 현재 스택

CloudSweep는 Claude Code-native 도구다.

```
런타임:     Claude Code (CLI / IDE extension)
스킬 정의:  SKILL.md (마크다운 규칙 + 탐지 조건)
규칙:       rules/*.json (임계값, 비용 fallback)
스크립트:   Python (parser, analyzer, formatter) — 일부 스킬
외부 연결:  aws-pricing MCP, aws-docs MCP
저장소:     로컬 파일 (finops_report.md, main_optimized.tf)
```

### 각 선택지 평가

#### Claude Subagent (현재 사용 중)

Agent tool로 서브에이전트를 직접 생성하는 방식. CloudSweep MA-001에서 지금 실행한 방식이 바로 이것이다.

| | 평가 |
|-|------|
| 통합 복잡도 | **없음** — 이미 작동 중 |
| 스킬 재사용 | SKILL.md를 프롬프트에 그대로 전달 |
| 인프라 요건 | 없음 |
| 한계 | Claude Code 세션 밖에서는 실행 불가. 스케줄 실행, API 트리거, 대화 간 상태 유지 불가 |

#### Strands (AWS Strands Agents)

Python SDK 기반 에이전트 프레임워크. `@tool` 데코레이터로 도구를 정의하고 에이전트가 이를 호출하는 구조다.

```python
from strands import Agent, tool

@tool
def analyze_lambda(tf_chunk: str, metrics_slice: str, cost_slice: str) -> str:
    # 기존 Python scripts/analyzer.py 로직 재사용 가능
    ...

agent = Agent(tools=[analyze_lambda])
```

| | 평가 |
|-|------|
| 통합 복잡도 | **중간** — SKILL.md를 Python 코드로 변환 필요. 기존 Python 스크립트는 그대로 재사용 |
| 스킬 재사용 | rules/*.json과 Python scripts 재사용 가능. SKILL.md는 시스템 프롬프트로 변환 |
| 인프라 요건 | Python 환경, Bedrock 접근 |
| 장점 | Lambda 스케줄 배포 가능. AWS 네이티브. CloudSweep의 AWS 분석 도구 성격과 정합 |
| 한계 | Bedrock 모델 전환 필요 (Claude Code 컨텍스트에서 Claude API로) |

#### LangGraph

그래프 기반 상태 머신으로 멀티 에이전트 워크플로를 정의한다.

| | 평가 |
|-|------|
| 통합 복잡도 | **높음** — 그래프 노드로 전체 재설계 필요 |
| 스킬 재사용 | SKILL.md 규칙 → 노드 함수 내 시스템 프롬프트, Python scripts → 노드 구현체 재사용 가능 |
| 장점 | Send API 동적 병렬 디스패치, Annotated reducer로 안전한 상태 병합, 조건부 루프, 체크포인팅 |
| CloudSweep 적합성 | **정적 Fan-out만이라면 과도. 동적 도메인 확장 + 체크포인팅을 원하면 적합** |

---

### LangGraph 전환 설계 — CloudSweep 모델 기준

#### LangGraph가 Agent tool보다 유리해지는 조건

현재 Agent tool 방식은 "dispatch 전에 모든 결정을 완료한다"는 전제 위에 동작한다. 다음 중 하나가 생기면 이 전제가 깨진다:

| 조건 | Agent tool로 처리 | LangGraph로 처리 |
|------|-----------------|-----------------|
| 워커 결과에서 새 도메인 발견 | 오케스트레이터 프롬프트에 재진입 (컨텍스트 팽창) | 조건부 엣지로 detect 노드 재실행 |
| 중간 findings 사람이 검토 후 계속 | 불가 (세션 단절) | MemorySaver 체크포인트로 일시 정지 |
| 도메인별 진행 상황 실시간 확인 | 불가 | 노드 단위 스트리밍 |
| 워커가 동일 상태 필드에 동시 쓰기 | 충돌 위험 (수동 병합 필요) | Annotated reducer가 자동 병합 |

#### 상태 스키마

LangGraph의 핵심은 `State`다. 모든 노드가 이 딕셔너리를 읽고 쓴다. `Annotated` 필드는 병렬 워커가 동시에 써도 안전하게 병합된다.

```python
from typing import TypedDict, Annotated
import operator

class CloudSweepState(TypedDict):
    # 입력 (불변)
    main_tf: str
    metrics_json: dict
    cost_report: dict

    # 오케스트레이터 제어 상태
    detected_domains: list[str]   # 지금까지 발견된 전체 도메인 (중복 방지)
    pending_domains: list[str]    # 이번 라운드에 분석할 도메인

    # 워커 결과 — operator.or_: 딕셔너리 병합 ({"lambda": ...} | {"s3": ...})
    domain_findings: Annotated[dict, operator.or_]

    # 확장 신호 — operator.add: 리스트 축적 (워커들이 동시에 append해도 안전)
    cross_domain_signals: Annotated[list, operator.add]

    # 최종
    final_report: str
```

#### 그래프 구조

```
Entry
  │
  ▼
[detect]──────────────────────────────────────────┐
  │  parse main.tf → pending_domains              │ 루프 (새 도메인 발견 시)
  │                                               │
  ▼  Send API: pending_domains 수만큼 병렬 생성   │
[analyze_domain] × N ──────────────────────────   │
  │  각 워커: domain_findings[domain] = findings  │
  │           cross_domain_signals += [...]       │
  │                                               │
  ▼  모든 워커 완료 후 자동 join                 │
[check_expansion]                                 │
  │  cross_domain_signals → 새 도메인 추출        │
  │                                               │
  ├──(새 도메인 있음)──────────────────────────────┘
  │
  └──(없음)──▶ [aggregate] ──▶ END
                  unified 4-section report
```

#### 구현 코드

```python
from langgraph.graph import StateGraph, END
from langgraph.constants import Send
from langgraph.checkpoint.memory import MemorySaver

# ── 노드 1: 도메인 감지 ──────────────────────────────────────────
def detect_node(state: CloudSweepState) -> dict:
    all_domains = parse_domains(state["main_tf"])
    already_done = set(state.get("detected_domains", []))
    new_domains = [d for d in all_domains if d not in already_done]
    return {
        "detected_domains": list(already_done) + new_domains,
        "pending_domains": new_domains,
        "cross_domain_signals": [],   # 이전 라운드 신호 초기화
    }

# ── 엣지: Send API로 동적 병렬 디스패치 ─────────────────────────
# detect 완료 후 pending_domains 수만큼 analyze_domain 노드를 동시 생성한다.
# Agent tool과 달리 타입 안전하고, 각 워커가 state를 공유하지 않는다.
def dispatch_to_analyzers(state: CloudSweepState) -> list[Send]:
    return [
        Send("analyze_domain", {
            "domain": domain,
            "tf_chunk": extract_tf_chunk(state["main_tf"], domain),
            "metrics_slice": slice_metrics(state["metrics_json"], domain),
            "cost_report": state["cost_report"],
        })
        for domain in state["pending_domains"]
    ]

# ── 노드 2: 도메인 분석 워커 (Send로 호출) ──────────────────────
def analyze_domain_node(worker_input: dict) -> dict:
    domain = worker_input["domain"]
    findings = run_finops_skill(
        domain=domain,
        tf_chunk=worker_input["tf_chunk"],
        metrics_slice=worker_input["metrics_slice"],
        cost_report=worker_input["cost_report"],
    )
    # 반환값은 Annotated reducer에 의해 메인 state에 병합된다
    return {
        "domain_findings": {domain: findings},
        "cross_domain_signals": findings.get("cross_domain_signals", []),
    }

# ── 노드 3: 확장 여부 결정 ───────────────────────────────────────
def check_expansion_node(state: CloudSweepState) -> dict:
    known = set(state["detected_domains"])
    new_domains = []
    for signal in state["cross_domain_signals"]:
        candidate = extract_domain_from_signal(signal)   # "DLQ → SQS" → "sqs"
        if candidate and candidate not in known:
            new_domains.append(candidate)
    return {"pending_domains": new_domains}

def should_loop_or_aggregate(state: CloudSweepState) -> str:
    return "detect" if state["pending_domains"] else "aggregate"

# ── 노드 4: 집계 ─────────────────────────────────────────────────
def aggregate_node(state: CloudSweepState) -> dict:
    report = build_unified_report(state["domain_findings"])
    return {"final_report": report}

# ── 그래프 조립 ──────────────────────────────────────────────────
builder = StateGraph(CloudSweepState)
builder.add_node("detect", detect_node)
builder.add_node("analyze_domain", analyze_domain_node)
builder.add_node("check_expansion", check_expansion_node)
builder.add_node("aggregate", aggregate_node)

builder.set_entry_point("detect")
builder.add_conditional_edges("detect", dispatch_to_analyzers, ["analyze_domain"])
builder.add_edge("analyze_domain", "check_expansion")   # 모든 워커 join 후 실행
builder.add_conditional_edges(
    "check_expansion",
    should_loop_or_aggregate,
    {"detect": "detect", "aggregate": "aggregate"},
)
builder.add_edge("aggregate", END)

# 체크포인터: aggregate 직전에 pause 가능 (human-in-the-loop)
graph = builder.compile(checkpointer=MemorySaver())
```

#### CloudSweep 현재 모델과의 대응

| 현재 (Agent tool / SKILL.md) | LangGraph 대응 |
|------------------------------|----------------|
| finops 오케스트레이터 프롬프트 | `detect_node` + `check_expansion_node` |
| `Agent(prompt=...)` 3개 동시 발사 | `Send("analyze_domain", ...)` × N |
| SKILL.md 도메인 규칙 | `run_finops_skill()` 내부 시스템 프롬프트 |
| 오케스트레이터가 결과 텍스트 수집 | `Annotated[dict, operator.or_]` 자동 병합 |
| 결과를 보고 새 도메인 수동 판단 | `check_expansion_node` + 조건부 루프 |
| metrics.json 도메인 슬라이싱 | `slice_metrics()` → `worker_input["metrics_slice"]` (동일) |

#### 전환 트리거 — 언제 LangGraph로 이동하나

지금 당장 전환할 이유는 없다. 다음 중 하나가 생기면 전환을 검토한다.

```
1. cross_domain_signals 루프가 실제로 필요해질 때
   예: Lambda 분석 중 SQS DLQ 발견 → SQS 도메인 자동 추가 분석

2. 분석 중간에 사람이 개입해야 할 때
   예: findings를 검토한 후 "이 도메인은 분석 생략" 결정

3. Claude Code 외부에서 실행해야 할 때
   예: FastAPI 엔드포인트로 CloudSweep 노출, GitHub Actions 스케줄 실행

4. 분석 진행 상황을 실시간으로 스트리밍해야 할 때
   예: 웹 UI에서 도메인별 완료 상태 표시
```

#### AgentCore (Amazon Bedrock AgentCore)

Bedrock Agents의 액션 그룹 기반 관리형 에이전트 런타임.

| | 평가 |
|-|------|
| 통합 복잡도 | **매우 높음** — 스킬을 Lambda 액션 그룹으로 전환, OpenAPI 스키마 작성, Bedrock Agent 등록 필요 |
| 인프라 요건 | Lambda, S3, Bedrock Agent, IAM 역할 등 |
| 장점 | 프로덕션 규모, 멀티 사용자, 관리형 메모리, 내장 guardrail |
| CloudSweep 적합성 | **현재 시점에서는 과도** — 프로덕션 배포가 목표가 되면 재검토 |

### 통합 용이성 순위

```
Claude Subagent  > Strands  > LangGraph  > AgentCore
    (이미 작동)    (중간 비용)  (높은 비용)   (최고 비용)
```

### 결론

**지금 단계**: Claude Subagent가 최적이다. CloudSweep는 Claude Code-native 도구이고, SKILL.md + Agent tool 조합은 이미 3/3 recall을 달성하고 있다. 추가 인프라 없이 작동한다.

**LangGraph로 이동할 때**: Claude Code 외부 실행, 동적 도메인 확장 루프, 체크포인팅이 필요해지는 시점이다. 위의 설계 그대로 이식 가능하며, 현재 SKILL.md 규칙과 Python scripts는 `run_finops_skill()` 내부에 그대로 재사용된다. Strands보다 LangGraph를 선택하는 이유는 Send API의 동적 병렬 디스패치와 조건부 루프가 CloudSweep의 확장 시나리오(동적 도메인 추가)에 더 직접적으로 맞기 때문이다.

AgentCore는 "멀티 사용자 프로덕션 서비스"가 되면 재검토한다.
