## 1. Option A — Multi-Agent SKILL.md 설계

`finops` orchestrator skill(`/.claude/skills/finops/SKILL.md`)을 수정해 멀티 도메인 병렬 디스패치를 지원하도록 확장했다.

### 핵심 변경점

| 변경 전 | 변경 후 |
|---------|---------|
| 단일 서비스만 라우팅 (가장 비용이 높은 하나 선택) | 도메인 감지 → 2개 이상이면 병렬 서브에이전트 디스패치 |
| 나머지 도메인 무시 | 각 에이전트가 자기 도메인 TF 청크 + 메트릭 슬라이스 + 비용 슬라이스만 수신 |
| 단일 `finops_report.md` | 도메인별 분석 → 오케스트레이터가 병합 + Cross-Domain 인터랙션 섹션 추가 |

### 멀티 에이전트 디스패치 흐름

```
finops orchestrator (SKILL.md)
│
├── main.tf 읽기 → 도메인 감지: lambda, s3, dynamodb
│
├── [병렬] Agent tool 3개 동시 발사
│     ├── Lambda 전문가: comp1_ TF chunk + Lambda 메트릭 + Lambda 비용
│     ├── S3 전문가:     comp2_ TF chunk + S3 메트릭 + S3 비용
│     └── DynamoDB 전문가: comp3_ TF chunk + DynamoDB 메트릭 + DynamoDB 비용
│
└── 결과 집계: finops_report.md + main_optimized.tf + Cross-Domain 섹션
```

**컨텍스트 격리 효과**: 각 전문가는 자기 도메인 리소스만 보므로 다른 도메인의 TF 노이즈 없이 집중 분석 가능.

## 4. 발견된 문제 전체 (3개 패턴)

### 패턴 1 — L2-014: Lambda 메모리 초과 프로비전

- **대상**: lambda-function-atrjum, 37781v, luzj7m
- **근거**: 30일 720 datapoints, memory_used_mb = 100.0 flat (분산 없음)
- **원인**: 3008MB 기본값(또는 복붙) 배포 후 power-tuning 미실시
- **수정**: `memory_size = 256` (2.5× headroom above p99 100MB)
- **절감**: **$750/mo** ($9,000/yr)

### 패턴 2 — L1-011: S3 Lifecycle 정책 부재

- **대상**: data-lake-raw (2.8TB), data-lake-archive (5.2TB)
- **근거**: Terraform에 `aws_s3_bucket_lifecycle_configuration` 블록 없음
- **원인**: 버킷 생성 시 lifecycle 거버넌스 미설정, archive 버킷이 명칭과 달리 Standard 유지
- **수정**: raw → STANDARD_IA(30d) + GLACIER(90d); archive → GLACIER(1d)
- **절감**: **$153/mo** ($1,836/yr)

### 패턴 3 — L1-010: DynamoDB 초과 프로비전

- **대상**: dynamodb-table-wouiv4
- **근거**: RCU 2% 활용 (100/5000), WCU 8.5% 활용 (85/1000), 30일 데이터
- **원인**: 피크 기준 프로비전 후 Auto Scaling 미설정, 실 트래픽은 안정적 저부하
- **수정**: read_capacity=120, write_capacity=120 + Auto Scaling (RCU max 600, WCU max 300)
- **절감**: **$522/mo** (보수적, $6,264/yr)

## 6. Cross-Domain 관찰

단일 에이전트는 이걸 볼 수 없다 — 오케스트레이터만 가능:

1. **Lambda → DynamoDB 연결 리스크**: Lambda timeout=30s 환경에서 DynamoDB를 right-size(120 RCU)하면, Auto Scaling 반응 지연 구간(scale-out cooldown 60s)에 읽기 throttle 발생 가능. Lambda retry 시 billed duration이 누적됨. → Auto Scaling max_capacity를 실제 피크의 5× 이상(600 RCU)으로 설정해 완충.
2. **Lambda 메모리와 DynamoDB SDK**: Python Lambda의 boto3 DynamoDB 클라이언트는 메모리를 적게 쓴다. 256MB로 줄여도 DynamoDB SDK 연산에 영향 없음.

| Single-Domain Routing 호출 방법 불명확 | 어떻게 sub-skill을 invoke하는지 미정의 | "Read `.claude/skills/finops-[service]/SKILL.md` directly" 명시 |
| Recall/토큰/wall-clock 측정 Outputs 없음 | hint 요구 4개 지표가 report에 미포함 | Required Measurement Block 섹션 추가 |

### 잔여 고려사항 (미수정, 구조적 한계)

- **토큰 15× 예측**: hint가 말한 15×는 metrics.json 원본 전달 시 기준. 통계 요약 전달로 3.1×로 줄였지만, 요약 과정에서 정보 손실 가능 (특히 burst 패턴 감지). L2 패턴처럼 시계열 분산이 중요한 경우 원본 필요 여부를 도메인별로 판단해야 함.
- **서브에이전트 출력 파일**: 서브에이전트가 WORK_DIR에 직접 파일을 쓰도록 할지, 텍스트 반환 후 오케스트레이터가 병합할지 — 현재 텍스트 반환 방식. 병렬 쓰기 시 파일 충돌 가능성 있어 텍스트 반환이 안전.

## 8. 회고

### 멀티 에이전트가 효과적인 조건
- 인프라가 2개 이상 서비스 도메인을 포함할 때
- 각 도메인의 메트릭/비용 데이터가 별개로 존재할 때 (컨텍스트 격리가 의미 있을 때)
- L2(30일 시계열) 패턴이 섞여 있어 단일 에이전트 컨텍스트를 압박할 때

### 멀티 에이전트의 비용
- 토큰 3.1× (메트릭 요약 전달 기준), 원본 metrics.json 전체 전달 시 ~15×
- 병렬 실행으로 wall-clock은 단일 대비 유사 또는 오히려 짧음
- 오케스트레이터 집계 단계 추가 구현 필요

### Recall 향상 정직성
- 베이스라인이 스스로 "나머지 두 도메인 분석 안 했다"고 인지했지만, 단일 라우팅 구조상 실행하지 않음
- 멀티 에이전트는 구조적으로 모든 도메인을 빠뜨릴 수 없음 (디스패치 루프가 전 도메인 대상)
- **정직한 결론**: 멀티 에이전트가 recall 측면에서 구조적으로 우월함. 토큰 비용은 실제 감수해야 할 트레이드오프.
