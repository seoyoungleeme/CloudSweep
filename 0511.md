# CloudSweep - 현황 및 발전 방향

**발표일: 2025-05-11**

---

## 1. 프로젝트 개요

CloudSweep은 AWS 클라우드 인프라의 비용 낭비를 탐지하고 최적화 방안을 제시하는 FinOps 분석 시스템이다.

분석 대상은 시뮬레이션 환경(ministack)에서 스캔된 리소스 데이터이며, 실제 인프라와 동일한 구조의 입력 파일을 기반으로 동작한다.

---

## 2. 현재 시스템 구조

### 2.1 입력 데이터

각 시나리오는 다음 파일들로 구성된다.

| 파일 | 내용 |
|------|------|
| `main.tf` | Terraform 인프라 정의 |
| `metrics/metrics.json` | 30일치 CloudWatch 지표 |
| `cost_report.json` | 6개월 비용 이력 |
| `business_metrics.json` | 비즈니스 컨텍스트 (L2 이상) |
| `tags_inventory.json` | 태그 현황 (L2 이상) |
| `ri_sp_coverage.json` | RI/SP 커버리지 (L3) |
| `cur_report.csv` | CUR 원본 데이터 (L3) |

시나리오는 L1~L3 난이도로 구성되며, L1은 단일 서비스 과잉 프로비저닝, L3는 조직 단위 할인 구조 분석까지 포괄한다.

### 2.2 분석 파이프라인

```
입력 파일
    ↓
finops (오케스트레이터 스킬)
    ↓ Terraform 리소스 타입 기반 라우팅
서비스별 분석 스킬
    ↓
finops_report.md + main_optimized.tf 생성
```

오케스트레이터가 `main.tf`에서 리소스 타입을 식별하고, 해당 서비스 스킬로 위임한다.
모든 분석은 Claude Code 스킬 레이어에서 수행된다.

### 2.3 지원 리소스 (15종)

| 스킬 | 탐지 대상 |
|------|-----------|
| finops-rds | 인스턴스 과잉 사이징, 불필요한 Multi-AZ, 엔진 Extended Support 비용 |
| finops-ebs | 고아 스냅샷, 과잉 보관 |
| finops-elb | 유휴 로드밸런서 |
| finops-s3 | 수명주기 정책 미비, 불완전 멀티파트 업로드 |
| finops-lambda | 메모리 과잉 할당, 타임아웃 과잉 설정 |
| finops-elasticache | 노드 과잉 사이징, 불필요한 레플리카 |
| finops-sqs | Short polling, 비효율적 배치 |
| finops-kinesis | 불필요한 Enhanced Fan-Out, 샤드 과잉 |
| finops-nat | S3/DynamoDB Gateway 엔드포인트 미사용 |
| finops-tgw | VPC Peering 대비 불필요한 Transit Gateway |
| finops-organizations | 통합 결제 미적용, RI/SP 공유 누락 |
| finops-cloudwatch | 로그 보관 정책 미설정 |
| finops-cloudwatch-alarm | 고해상도 알람 불필요 사용 |
| finops-dynamodb | 용량 모드 미최적화 |
| finops-ecs | Fargate 태스크 과잉 사이징 |

### 2.4 MCP 서버 활용

| MCP | 용도 |
|-----|------|
| aws-docs | 분석 결과에 근거 설명 추가 (공식 문서 인용) |
| aws-pricing | 실제 AWS 단가 조회 (절감액 산출 근거) |

분석 완료 시 각 시나리오 디렉터리에 생성된다.

## 4. 발전 방향

### 4.1 오케스트레이션 레이어: LangGraph 도입

현재는 Claude Code 스킬 간 단순 위임 구조다. 이를 LangGraph 기반 에이전트 그래프로 전환한다.

각 역할을 독립 에이전트로 분리한다.

| 에이전트 | 책임 |
|----------|------|
| 수집 에이전트 | 대상 환경에서 인프라 데이터 수집 |
| 가격 조회 에이전트 | AWS Pricing API 단가 조회 |
| 리소스 분석 에이전트 | 서비스별 낭비 탐지 (현재 스킬 역할) |
| 리포트 생성 에이전트 | 분석 결과 취합 및 보고서 작성 |

에이전트 간 상태는 LangGraph 그래프로 명시적으로 관리하며, 병렬 분석 및 조건 분기가 가능해진다.

### 4.2 리포트 생성: 로컬 LLM 분리

현재 리포트 생성까지 Claude Code에서 수행하고 있다. 리포트 생성 단계는 로컬 LLM으로 분리하여 처리한다.

- Claude Code: 근거 기반 분석, 판단, 수치 계산
- 로컬 LLM: 분석 결과를 정형화된 리포트 문서로 변환

비용 절감과 함께 리포트 생성 파이프라인의 독립성을 확보한다.

### 4.3 Cross-Service 결합 낭비 분석

현재는 서비스별 단독 분석만 수행한다. 이후에는 복수 서비스 간의 결합 낭비 패턴을 탐지한다.

예시:
- NAT Gateway 트래픽 증가 + S3 접근 패턴 → Gateway 엔드포인트 미사용 낭비
- ECS Fargate 과잉 메모리 + Lambda 재시도 급증 → 아키텍처 수준 비효율

여러 분석 결과를 결합해 단일 서비스 관점에서는 보이지 않는 낭비를 탐지하는 것이 목표다.

### 4.4 패턴 학습 및 다중 패턴 분석

반복 탐지된 낭비 패턴을 누적하고, 새로운 시나리오 분석 시 기존 패턴과의 유사도를 활용한다.

시나리오 유형별 패턴 라이브러리를 구축하고, 유사 패턴 검토를 분석 과정에 포함시켜 탐지 정확도를 높인다.

## 5. 로드맵 요약

```
현재
└── Claude Code 스킬 기반 단일 파이프라인 (15종 리소스 분석)

단기
└── LangGraph 에이전트 구조 전환
    └── 수집 / 가격 / 분석 / 리포트 에이전트 분리
    └── 리포트 생성 → 로컬 LLM

중기
└── Cross-Service 결합 낭비 탐지
    └── 복수 서비스 간 낭비 패턴 분석

장기
└── 패턴 라이브러리 구축
    └── 다중 패턴 학습 기반 자동 분석
```
