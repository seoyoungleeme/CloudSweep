# 0601 — LV-001: Cost Spike Detection 회고

## 1. 문제 상황

**시나리오 요약**: 프로덕션 환경에서 약 25시간 전부터 비정상적인 비용 증가가 감지됨. AWS Cost Explorer의 시간별 데이터를 분석해 spike 원인을 자동으로 파악하는 에이전트를 구축하는 과제.

**기존 finops 스킬과의 차이점**이 핵심 문제였다.

기존 스킬들은 모두 **Terraform 인프라 분석** 기반이었다:
- `main.tf` (리소스 정의) + `metrics.json` (CloudWatch 지표) + `cost_report.json` (비용 리포트)

LV-001은 구조 자체가 달랐다:
- **Terraform 없음** — 배포된 인프라 정의 파일 미제공
- **Cost Explorer API mock 응답** — `mock_responses/get_cost_and_usage_hourly.json`, `mock_responses/get_anomalies.json`
- 입력이 "현재 인프라 스펙"이 아니라 "실시간 비용 시계열 데이터"

즉 기존 finops 오케스트레이터를 그대로 실행하면 `main.tf`를 찾다가 실패하거나 잘못 라우팅될 수 있는 상황이었다.

**mock 데이터 형태**:
- `get_cost_and_usage_hourly.json`: Cost Explorer `HOURLY` granularity 응답 형식이지만, 실제로는 5개 sampled datapoint만 제공됨 (2026-05-26 00:00Z ~ 2026-05-27 06:00Z). 연속 hourly 시계열이 아니므로 baseline 신뢰도는 Medium으로 낮춰 표기했다.
- `get_anomalies.json`: EC2 (score 0.88, impact $312.45), S3 (score 0.65, impact $78.60) 2건 anomaly

## 2. 예상 해결방안

hint.txt에서 제안한 접근법:

1. **Spike 감지**: rolling average + 표준편차 기반 통계 flagging, 이전 datapoint 대비 pct-change 보조 신호
2. **드릴다운**: SERVICE별 GroupBy → USAGE_TYPE별 세부 분해 → LINKED_ACCOUNT 다차원 분석
3. **CloudTrail 상관**: spike 발생 30~60분 전 이벤트 탐색, RunInstances / CreateAutoScalingGroup 등 highlight
4. **파이프라인 구조**: `detect() → drilldown() → correlate() → report()`
5. **페이지네이션**: GetCostAndUsage는 1일 HOURLY 데이터만 반환 → NextPageToken 처리

## 3. 실제 해결방법

### 3-1. finops 오케스트레이터 라우팅 확장

finops SKILL.md에 **Evidence-Driven Routing**을 추가했다. 처음에는 `Terraform mode`와 `Cost Explorer Anomaly Mode`를 닫힌 분기로 나누는 방식으로 생각했지만, 실제 환경에서는 비용 spike 분석과 Terraform remediation이 함께 필요한 경우가 많다. 그래서 최종 설계는 "모드 선택"이 아니라 **사용 가능한 증거를 먼저 inventory하고 필요한 analyzer를 조합하는 방식**으로 정리했다.

라우팅은 세 단계로 생각한다:

1. `evidence_available`: CE time-series, GetAnomalies, CloudTrail, Terraform, metrics, cost_report 중 무엇이 있는지 확인
2. `analysis_intent`: cost spike incident / waste optimization / blended 중 어떤 질문인지 판단
3. `execution_plan`: 필요한 analyzer node를 조합

| 증거 / 의도 | 실행 계획 |
|-------------|-----------|
| CE time-series 또는 GetAnomalies만 있고 Terraform 없음 | `finops-anomaly` standalone 실행, `main.tf` 요구하지 않음 |
| `get_cost_and_usage*.json` 존재 | full anomaly path: spike 감지 + drilldown + anomaly confirmation |
| `get_anomalies.json`만 존재 | degraded anomaly path: anomaly 요약만 생성, spike 감지 스킵 |
| Terraform만 존재 | 기존 Domain Detection + service-specific FinOps routing |
| CE incident evidence + Terraform 공존 | `finops-anomaly` 먼저 실행 후 Domain Detection 계속 수행. spike window, top service, UsageType, CloudTrail gap을 `RELATED_CONTEXT`로 넘겨 domain analyzer와 remediation에 활용 |

즉 LV-001은 CE evidence만 있으므로 anomaly standalone으로 끝나지만, 실제 운영 환경에서는 `CE spike → CloudTrail correlation → Terraform resource mapping → IaC remediation`까지 이어지는 blended execution plan이 가능해야 한다.

### 3-2. finops-anomaly 스킬 신설

`.claude/skills/finops-anomaly/SKILL.md` 전체 신규 작성.

4단계 워크플로우:
1. **detect()**: CE 시계열 파싱 → rolling baseline → statistical + pct-change dual flag
2. **drilldown()**: 서비스별 delta 순위 → GetAnomalies RootCauses 교차 참조 → UsageType attribution
3. **correlate()**: CloudTrail mock 탐색 → spike 전 60분 이벤트 필터링
4. **report()**: `result/finops_report.md` 생성

solution.py 생성 규격 명세 포함 (6개 함수 인터페이스 고정).

### 3-3. solution.py 구현

표준 라이브러리만 사용 (boto3는 `--live` 모드에서만 선택적 import).

```
load_cost_and_usage()  →  detect_spikes()  →  drilldown_services()
                                                       ↓
render_report()  ←  correlate_cloudtrail()
```

mock 모드: `mock_responses/*.json` 파일 읽기  
live 모드: `--live` 플래그 → boto3 `ce.get_cost_and_usage` + `ce.get_anomalies` 직접 호출  
페이지네이션: `NextPageToken` 루프로 처리 (live 모드)

## 5. 중점적으로 신경 쓴 것

### 5-1. Evidence-first rule 적용

CloudTrail mock이 없으면 triggering event에 대해 **추정하지 않는다**. Confidence 표도 고정 문구가 아니라 실제 분석 결과에서 파생한다. 최종 산출 표:

```
Confidence 테이블 (데이터 기반 산출):
- Spike 감지: High      (통계 + pct-change 두 flag 모두 발화)
- Service attribution: Medium  (EC2·S3는 confirmed, RDS는 GetAnomalies에 없어 Suspected)
- Triggering event: Low   (CloudTrail 이벤트 없음 — instrumentation gap)
- Baseline reliability: Medium  (sparse window, prior datapoint 3개)
```

특히 Service attribution은 EC2·S3처럼 GetAnomalies로 확인된 항목과 RDS처럼 의심 수준에 머무는 항목을 분리해 계산한다. 드릴다운 3위 RDS가 GetAnomalies에 없는데도 High라고 단언하면 evidence-first 원칙 위반이다. 보고서에서 낮은 confidence를 숨기지 않고 표에 명시하는 것이 핵심이었고, 시나리오 파일명이나 hint.txt 내용을 "증거"로 쓰지 않는다.

### 5-2. Dual-flag spike 감지

단일 신호에만 의존하지 않도록 두 가지 flag를 병렬 적용:

- **통계 flag**: `cost > baseline_mean + 2 * baseline_stddev` (최근 최대 6개 datapoint rolling baseline, n≥3 필요)
- **pct-change flag**: 이전 datapoint 대비 `>100%` 증가

둘 중 하나라도 발화하면 spike로 판정. mock에서는 두 flag 모두 발화 (`$58.74 > $13.33`, `+351.5%`).

중요한 점은 구현이 "최근 6시간"이 아니라 **현재 datapoint 이전 최대 6개 datapoint**를 baseline으로 사용한다는 것이다. LV-001 mock은 연속 hourly 데이터가 아니어서 spike 직전 baseline은 3개 datapoint만 사용했다. 따라서 pct-change도 "이전 시간"이 아니라 "이전 datapoint" 대비 변화율로 계산했다.

### 5-3. Service 이름 정규화

CE `GetCostAndUsage`와 `GetAnomalies`가 서비스 이름을 다르게 반환한다는 함정이 있었다:

```python
SERVICE_ALIASES = {
    "Amazon S3": "Amazon Simple Storage Service",
    "Amazon EC2": "Amazon Elastic Compute Cloud",
    ...
}
```

drilldown에서 GetAnomalies RootCauses를 교차 참조할 때 이 alias 매핑으로 정규화해야 UsageType을 제대로 가져올 수 있다. 이 부분 없으면 S3 → "Not in GetAnomalies - Suspected" 처리됨.

### 5-4. finops 오케스트레이터 라우팅 우선순위

CE mock 파일이 있으면 Terraform 파일이 없어도 analysis를 거부하지 않도록 설계했다. 다만 CE evidence가 있다는 이유만으로 Terraform/domain 분석을 항상 skip하지는 않는다. 실제 환경에서 비용 spike는 배포/오토스케일링/Terraform 설정과 이어질 수 있기 때문이다.

최종 규칙은 다음과 같다:

```text
if CE evidence exists:
    run finops-anomaly

if Terraform exists:
    continue Domain Detection and service analyzers
    pass anomaly summary as RELATED_CONTEXT when available

if CE exists and Terraform is absent:
    anomaly standalone path
```

이렇게 하면 LV-001처럼 CE만 있는 샘플도 처리하고, 향후 실제 AWS 환경에서 CE + CloudTrail + Terraform을 동시에 가진 incident-remediation 흐름도 같은 orchestrator 안에서 처리할 수 있다.

### 5-5. GetAnomalies 응답 키 확인

GetAnomalies mock의 top-level key는 `Anomalies`였다. 설계 중 AWS 응답 형태를 추정해서 `AnomalyList` 같은 이름을 쓰면 `RootCauses`, `Impact`, `AnomalyScore`가 전부 비게 된다. 실제 fixture를 열어서 key를 확인하고 `solution.py`의 `load_anomalies()`가 `response["Anomalies"]`를 읽도록 맞춘 것이 중요했다.

## 6. 부족한 것 / 보완할 방법

### 6-1. CloudTrail mock 부재 → triggering event 미확인

가장 큰 공백. EC2 비용이 $8.60 → $45.30으로 증가했고 c5.4xlarge UsageType이 명확히 식별됐음에도 "무엇이 트리거했는가"는 답하지 못한다.

**보완**: spike 이전 60분 창을 커버하는 CloudTrail mock 추가 (`mock_responses/cloudtrail.json`). 실제 환경이라면 `cloudtrail:LookupEvents`를 `startTime=spike_start-60m`으로 호출하되, EC2 비용 spike라고 해서 `ec2.amazonaws.com`만 보면 부족하다. ASG scale-out, ECS service update, CloudFormation stack update가 EC2 비용을 유발할 수 있으므로 `ec2.amazonaws.com`, `autoscaling.amazonaws.com`, `ecs.amazonaws.com`, `cloudformation.amazonaws.com`을 함께 조회해야 한다.

### 6-2. 시계열 데이터 sparse (5개 datapoint)

mock은 Cost Explorer `HOURLY` 응답 형식이지만 연속 hourly 시계열이 아니라 5개 sampled datapoint만 포함한다. spike 직전 rolling 6-point window가 실제로 3개 prior point밖에 못 쓰고, stddev가 $0.46로 매우 좁아서 spike가 극단적으로 보이는 효과가 있다. 실제 운영에서는 연속 hourly 데이터 (24~72개)를 써야 baseline이 신뢰성 있다.

**보완**: 연속 hourly datapoint mock 파일 추가하거나, mock이 sparse할 때 baseline reliability를 "Low"로 강등하는 조건 추가.

### 6-3. UsageType 2nd-pass 드릴다운 미구현

SKILL.md에는 "UsageType별 두 번째 GetCostAndUsage 호출 (GroupBy=USAGE_TYPE)" 명세가 있지만 mock에 usagetype mock 파일이 없어서 GetAnomalies.RootCauses에 의존했다. RDS처럼 GetAnomalies에 없는 서비스는 UsageType 미확인("Suspected") 처리됐다.

**보완**: `mock_responses/get_cost_and_usage_usagetype_ec2.json` mock 파일 추가 + `drilldown_services()` 함수에서 usagetype mock를 먼저 찾고, 없으면 RootCauses fallback하는 로직 구현.

## 7. LangGraph 도입 — 현재 단계와 한계

### 7-1. 왜 Step 1(선형 그래프)부터 했는가

솔직히 말하면 Step 1 단계의 LangGraph는 **없어도 동작이 동일하다.** 기존 `main()`이 함수를 순서대로 호출하던 것을 StateGraph 노드로 감싼 것뿐이고, 선형 파이프라인에서 LangGraph가 추가로 주는 것은 없다.

그럼에도 Step 1을 먼저 한 이유는 하나다: **Step 2(LLM 노드)와 Step 3(조건 분기)를 붙이기 전에 노드 경계와 State 구조를 먼저 확정하기 위한 scaffolding.** 기존 6개 함수를 노드로 분리하는 작업을 나중에 하면 LLM·분기 로직과 뒤섞여 복잡해진다. Step 1은 그 자체로 가치 있는 게 아니라 다음 단계를 깔끔하게 만들기 위한 준비다.

### 7-2. Step 2 — report 노드에만 LLM 도입

LangGraph를 쓰는 실질적 이유가 처음 생기는 단계다. `_node_report`가 Claude API를 호출해 Section 6(Root Cause)과 Section 7(Remediation)을 생성한다. 나머지 노드(load/detect/drilldown/correlate)는 모두 deterministic Python 계산이라 LLM이 필요 없다.

세 겹 폴백으로 LLM 없어도 보고서는 항상 생성된다:
```
① API 키 없음/호출 실패 → 템플릿 폴백
② 빈 응답              → 템플릿 폴백
③ 구조 검증 실패        → 템플릿 폴백 (_valid_llm_sections)
```

`temperature=0`으로 재현성 확보, `_valid_llm_sections()`로 헤더 누락·중복·번호 침범 방지.

### 7-3. 가장 큰 변화는 아직 안 했다

현재까지 건드린 건 `solution.py` — 즉 특정 시나리오의 분석 코드다. **SKILL.md 오케스트레이터는 그대로다.** 진짜 "Claude Code CLI 의존 제거"는 SKILL.md의 라우팅 로직(evidence-driven routing, domain detection, multi-domain dispatch)이 코드로 내려오는 Step 3부터다. 그 단계에서:

- finops orchestrator의 라우팅 규칙 → conditional edge
- SKILL.md의 evidence-first rule → system prompt
- 멀티도메인 병렬 dispatch → `Send` API

가 돼야 비로소 `/finops` 없이 `python pipeline.py`로 독립 실행이 가능해진다. 지금은 `solution.py`만 LangGraph 위에 있고, 그 위 오케스트레이터는 여전히 Claude가 SKILL.md를 읽고 실행하는 구조다.

## 공유할 만한 포인트

### "Terraform 없음"이 라우팅 실패를 일으킬 수 있다

finops 오케스트레이터가 처음에는 항상 main.tf를 찾도록 설계돼 있었다. CE API 응답이 입력인 시나리오는 애초에 Terraform이 없고, 없어도 맞는 경우다. 입력 형태를 먼저 판별하는 라우팅 우선순위가 중요했다.

### CE 서비스 이름 불일치는 교과서적 함정

GetCostAndUsage Groups는 "Amazon S3"를 쓰고 GetAnomalies DimensionValue는 "Amazon Simple Storage Service"를 쓴다. alias 없이 직접 비교하면 S3 cross-reference가 조용히 실패한다. AWS API 전반에서 동일 서비스의 명칭이 endpoint마다 다른 경우가 많다.

### 스파이크 감지는 단일 신호만으로 불안정하다

통계 flag 단독으로 쓰면 데이터가 sparse할 때 threshold가 너무 좁아지고, pct-change 단독으로 쓰면 절대 금액이 작아도 flag가 발화한다. 두 신호를 OR로 합쳐서 coverage를 넓히되, 보고서에 "어떤 flag가 발화했는가"를 명시해 false positive 판별을 사람이 할 수 있게 남겨두는 것이 핵심이다.

### evidence-first는 신뢰도 표를 위한 설계다

CloudTrail 없이도 보고서를 완성할 수 있지만, 그 공백을 "Low confidence"로 명시해 받는 사람이 다음 액션(CloudTrail 계측 추가)을 알 수 있게 해야 한다. 공백을 채우기 위해 hint나 시나리오 이름에서 추론하는 것은 보고서의 신뢰성을 망가뜨린다.
